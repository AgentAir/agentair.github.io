Azure BastionとNSGとJIT Network Accessで混乱したこと

2026-04-30 / Azure / Bastion / NSG

Home Articles About

はじめに

Azure VMへ安全に接続する方法を調べている中で、Azure Bastion、NSG、JIT Network Accessという 3つの機能が頻繁に登場し、それぞれの関係を整理するのに時間がかかりました。

最初はこれらをまとめて「VM接続の許可設定」として同じ層の話と捉えてしまっていましたが、 実際にはそれぞれ役割が異なります。 本記事は、混乱した経験をもとに3つの機能を分けて整理した個人の学習メモです。 公式ドキュメントの代替ではなく、自分なりの理解を記録することを目的としています。

Azure Bastionとは何か

Azure BastionはAzure VMへRDP/SSH接続するための、Azureが管理するマネージドサービスです。 VMにパブリックIPアドレスを付けることなく、Azureポータルのブラウザ経由、 またはSKUや構成によってはネイティブクライアント（mstsc、OpenSSHなど）経由で接続できます。

自分の理解では、Bastionは「接続経路を提供するもの」です。 VMに直接インターネットからRDP/SSHを開けるのではなく、Bastion経由という一本の管理経路を通じて接続する構成になります。

オンプレミス運用で「管理用の踏み台サーバー（Linux踏み台）」を立てて、 そこ経由でSSHするという構成に近いイメージがあります。 ただし、自分でサーバーを管理・更新・監視する必要がない点が、マネージドサービスとしての大きな違いです。

Bastionを使う場合、VM自体にパブリックIPを持たせない設計が基本です。 この点が、VMの接続セキュリティを考える上で重要な前提になります。

NSGとは何か

NSG（Network Security Group）はAzureのサブネットまたはNICに関連付けて、 インバウンド・アウトバウンドの通信を許可・拒否するための仕組みです。 オンプレミスのACL（アクセスコントロールリスト）に近い役割を持ちます。

オンプレミスのACLとの違いとして、Azureには既定のシステムルール（デフォルトルール）があり、 これを上書きする形で独自のルールを追加します。 ルールには優先度（数値が小さいほど優先）があり、先に一致したルールが適用されます。

Bastionを使う場合でも、NSGはVMのNICやサブネットに存在する通信制御として常に有効です。 Bastion経由で接続していても、VM側のNSGがRDP（3389）やSSH（22）へのインバウンドを拒否していれば、 接続できない場合があります。 また、Bastionサブネット（AzureBastionSubnet）にも専用のNSGルールが必要な場合があります。

JIT Network Accessとは何か

JIT（Just-In-Time）Network AccessはMicrosoft Defender for Cloudの機能です。 RDPやSSHなどの管理ポートを常時インターネットに開放するのではなく、 必要な時間だけ、必要な送信元IPアドレスに対して一時的にアクセスを許可するという考え方です。

JITが有効になると、Defender for CloudがNSGやAzure Firewallのルールを動的に操作します。 アクセス要求を承認するとJIT用のNSGルール（MicrosoftDefenderForCloud-JITRule のような名前）が 自動で追加・削除されます。 このルールが自動で出てくることを知らなかったため、最初は「誰がこのNSGルールを作ったのか」と混乱しました。

JITの主な対象は、VMにパブリックIPがあり、管理ポートがインターネットに開きうる構成です。 JITを使うことで、常時開放を避けながら必要なときだけアクセスを許可するという運用が実現できます。

自分が混乱したポイント

3つの機能を理解する前に、次のような疑問が重なって混乱していました。

  • BastionとJITは両方必要なのか： Bastionを使えばVMにパブリックIPが不要になる。 ではJITは不要になるのか、それとも組み合わせて使うものなのか、最初はわからなかった。
  • BastionがあってもNSGでRDP/SSHを開ける必要があるのか： Bastionは経路を提供するが、VM側のNSGでの許可が必要かどうかが曖昧だった。 結果として、Bastionサブネットへの通信とVMへの通信はそれぞれ独立してNSGで制御されることを理解した。
  • VMにパブリックIPがない場合、JITの意味はどう考えるのか： JITは管理ポートの常時開放を防ぐためのものだが、 そもそもパブリックIPがなければインターネットから管理ポートを開けようがない。 この場合、JITの優先度は下がると理解した。
  • MicrosoftDefenderForCloud-JITRule が突然NSGに現れた： Defender for CloudのJIT機能が有効になっていると、NSGにJIT用のルールが自動で追加・削除される。 これを知らずにNSGを確認すると、見覚えのないルールが出てきて原因がわからず混乱した。
  • Bastion、NSG、JITを「同じ層の設定」として見てしまった： それぞれが異なるレイヤーの機能であることを理解する前に、まとめて「アクセス制御」として見ていたため、 どこを変更すれば何が変わるのかが把握できていなかった。

自分なりの整理

3つの機能を役割で分けると、自分の中では次のように整理できました。 あくまでも学習段階での理解です。

  • Azure Bastion：「接続経路」を提供する
    VMにパブリックIPを持たせずにRDP/SSHできる経路を作る。 Bastionサブネット（AzureBastionSubnet）にデプロイされ、ブラウザやネイティブクライアントから接続できる。
  • NSG：「通信制御」を行う
    サブネットまたはNICに付与する通信フィルタリングの仕組み。 Bastionサブネット向けのNSGと、VM向けのNSGはそれぞれ独立して確認が必要。
  • JIT Network Access：「管理ポートの常時開放を防ぐ運用の仕組み」
    Defender for Cloudが管理する機能で、NSGルールを動的に操作して管理ポートを必要な時間だけ開放する。 VMにパブリックIPがある構成で特に有効。

VMへ接続できないトラブルが発生したときは、「経路」「通信制御」「セキュリティ運用」の どのレイヤーで止まっているかを分けて確認することが、切り分けの基本だと理解しました。

切り分けの考え方

VMへの接続が失敗したとき、自分が確認する順序を整理しました。

  1. VMにパブリックIPがあるかを確認する： パブリックIPがない場合、インターネットから直接接続することはできない。 Bastion経由またはVPN/ExpressRoute経由が前提になる。
  2. Bastion経由で接続しているかを確認する： AzureBastionSubnetにBastionがデプロイされているか、 SKUと構成が接続方法（ブラウザ/ネイティブクライアント）に対応しているかを確認する。
  3. NSGで必要な通信が許可されているかを確認する： Bastionサブネット向けのインバウンドルール（GatewayManagerからの443など）と、 VMサブネットまたはNIC向けのインバウンドルール（BastionサービスタグからのRDP/SSH）が正しく設定されているかを確認する。
  4. JITが有効でアクセス要求が必要な状態かを確認する： Defender for CloudでJITが有効な場合、アクセス要求を送って承認されなければNSGルールが開かない。 NSGにJITルールが存在するかを確認する。
  5. Defender for CloudのJITルールがNSGに影響していないかを確認する： MicrosoftDefenderForCloud-JITRule のようなルールが意図せず拒否方向で残っている場合がある。
  6. OS側のファイアウォールやRDP/SSHサービスの状態を確認する： NSGが正しく設定されていても、OS側のWindows ファイアウォールやsshd（SSH daemon）が 動いていなければ接続できない。 Azureポータルの「実行コマンド」機能やシリアルコンソールで確認できる場合がある。

学んだこと

  • 「経路」「通信制御」「セキュリティ運用」を分けて考える： Bastionは経路、NSGは通信制御、JITはセキュリティ運用の仕組みという整理をしておくと、 トラブル時にどこを確認すべきかが明確になる。
  • BastionはVMのパブリックIP不要を実現する経路： Bastionがあることでインターネットから直接VMのRDP/SSHポートを開けなくて済む。 ただし、Bastionサブネットへの通信やVM側のNSGは別途設定が必要。
  • JITはBastionとは独立した機能： JITはDefender for Cloudの管理ポート保護機能であり、Bastionの有無に関わらず機能する。 VMにパブリックIPがない構成ではJITの優先度は下がる場合があるが、 パブリックIPを付ける場合には有効な防御手段になる。
  • NSGは常に確認対象： BastionやJITの設定が正しくても、NSGのルールが原因で通信が止まる場合がある。 NSGの有効なルール（実効ルール）をAzureポータルで確認する手順を覚えておくと切り分けが速くなる。

まとめ

Azure Bastion、NSG、JIT Network Accessは、いずれもVM接続のセキュリティに関わりますが、 それぞれ役割が異なります。 Bastionは接続経路、NSGは通信制御、JITは管理ポートの常時開放を防ぐ運用の仕組みとして分けて理解することで、 混乱が減りました。

VMへの接続トラブルが発生したときは、「経路」「通信制御」「セキュリティ運用」のどのレイヤーで 止まっているかを分けて確認することが、切り分けの基本だと感じました。

今後はBastion、NSG、JITを組み合わせた最小構成を実際に検証し、 それぞれの設定が通信にどう影響するかを手を動かして確認していきたいと考えています。
