Azure Private EndpointでStorageへプライベート接続する検証メモ

2026-04-30 / Azure / Private Endpoint / Storage / Study Log

Home Articles About

はじめに

Azure Storageへインターネット経由ではなく、Private Endpoint経由でプライベート接続する検証を行いました。 本記事では、ネットワークエンジニア目線で混乱しやすかったポイントを整理します。

本記事は個人の検証メモです。会社名、顧客名、実IPアドレス、サブスクリプションID、SAS URLなどの機密情報は記載しません。

やりたかったこと

Azure VMからAzure Storageへアクセスする際に、パブリックインターネットを経由せず、 VNet内のプライベートIPアドレスを使って接続できることを確認したかったです。

検証イメージは以下です。

Azure VM
  ↓
VNet / Subnet
  ↓
Private Endpoint
  ↓
Azure Storage Account

検証構成

  • Azure VMを1台用意
  • Azure Storage Accountを作成
  • Storage AccountにPrivate Endpointを作成
  • VMとPrivate Endpointは同じVNet、または通信可能なVNetに配置
  • Private DNS Zoneを使ってStorageのFQDNをプライベートIPへ名前解決

ここで重要なのは、Private Endpointを作成するだけではなく、 DNSによってStorageのFQDNがPrivate EndpointのプライベートIPへ解決される必要がある点です。

Private Endpointで混乱したポイント

1. Private EndpointはStorageそのものではない

Private Endpointは、Azure StorageなどのPaaSサービスへプライベートIPで接続するためのネットワークインターフェースのような存在です。 Storage Account自体がVNet内に配置されるわけではなく、VNet内に接続口が作られるイメージです。

2. Blob / File / Queue / Table のどれに接続するかを選ぶ必要がある

Azure StorageにはBlob、File、Queue、Tableなど複数のサービスがあります。 Private Endpoint作成時には、どのサブリソースに対してPrivate Endpointを作るかを選択します。

例えばBlobを検証したい場合はBlob用のPrivate Endpoint、File Shareを検証したい場合はFile用のPrivate Endpointが必要です。 ここを間違えると、意図したサービスへプライベート接続できません。

3. 名前解決が重要

Private Endpointの検証では、IPアドレスだけでなくFQDNの名前解決を確認する必要があります。 Storage AccountのURLにアクセスしたとき、パブリックIPではなくPrivate EndpointのプライベートIPへ解決されることが重要です。 

nslookup <storage-account-name>.blob.core.windows.net

この結果がPrivate EndpointのプライベートIPを返していれば、DNSの観点では期待した状態に近いです。

確認したこと

1. VMから名前解決を確認

まずAzure VM上で、Storage AccountのFQDNがどのIPアドレスに解決されるか確認しました。 

nslookup <storage-account-name>.blob.core.windows.net

Private EndpointのプライベートIPが返ってくる場合、少なくとも名前解決はPrivate Endpoint向けになっています。

2. VMからStorageへアクセス

次に、VM上のブラウザやcurlなどでStorageへアクセスし、通信できるか確認しました。 

curl -I "https://<storage-account-name>.blob.core.windows.net/<container-name>/<file-name>?<SAS-token>"

ここではSAS URLを使う場合でも、SASトークンそのものは絶対に公開しないように注意します。

詰まったエラー

AuthenticationFailed

SAS URLでアクセスした際に、認証に失敗することがあります。 この場合、Private Endpointの問題ではなく、SASの権限、対象リソース、期限、署名などが原因の可能性があります。

PublicAccessNotPermitted

Storage Account側でパブリックアクセスが禁止されている場合に表示されることがあります。 これは設定としては正しい場合もあり、Private Endpoint経由でアクセスできているかを切り分ける必要があります。

AuthorizationPermissionMismatch

SASに必要な権限が不足している場合に発生することがあります。 例えば読み取りだけでよいのか、一覧表示も必要なのか、Blob単位なのかContainer単位なのかを確認する必要があります。

学んだこと

  • Private EndpointはPaaSサービスをVNet内に置く機能ではなく、プライベート接続口を作る機能として理解するとよい
  • Blob、File、Queue、Tableなど、対象サブリソースを意識する必要がある
  • Private Endpointの検証では、通信確認だけでなく名前解決の確認が重要
  • SASエラーはPrivate Endpointではなく、権限やスコープの問題であることも多い
  • ネットワーク、DNS、認証を分けて切り分けると理解しやすい

まとめ

Azure Private Endpointは、オンプレミスのネットワーク機器だけを触ってきた感覚だと最初は少し分かりづらいです。 特に、PaaSサービス、DNS、認証、SAS、Storageのサブリソースが絡むため、単純な疎通確認だけでは判断できません。

今後は、Private Endpoint、Private DNS Zone、Storage Firewall、SAS、RBACの関係も整理していきたいと思います。
