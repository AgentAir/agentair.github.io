TerraformでAzure VNetを作る前に理解したいこと

2026-04-30 / Terraform / Azure / VNet

Home Articles About

はじめに

前回の記事「Terraformとは何かをネットワークエンジニア目線で整理する」では、 provider・resource・state・plan・apply という基本概念を整理しました。

今回はその続きとして、TerraformでAzure VNetを実際に作る前に理解しておきたいことを整理します。 コードを書き始める前に、どんな要素を設計・把握しておくべきかを確認するのが目的です。

本記事は公式ドキュメントの代替ではなく、ネットワークエンジニアとしての視点で整理した個人の学習メモです。

TerraformでAzureを操作するにはProviderが必要

TerraformがAzureのリソースを作成・変更・削除するには、azurerm プロバイダーが必要です。 プロバイダーはTerraformとAzureの間の接続口のような役割を担っており、 Azure Resource Manager APIを通じてリソースを操作します。

ネットワーク機器の運用に例えると、SSHで機器に接続するための接続設定（IPアドレス、認証情報、プロトコル）を 事前に定義するのに近いイメージです。 Terraformでもプロバイダー設定をコードに書いておくことで、どのAzureテナント・サブスクリプションに対して操作を行うかを明示します。

認証方法はいくつかあります（Azure CLIログイン、サービスプリンシパルなど）が、 まず個人検証環境で試す場合はAzure CLIで事前にログインしておく方法が手軽です。 認証情報やシークレットをコードに直接書くのは避け、環境変数や外部ファイルで管理するのが原則です。

VNet作成時に考える要素

TerraformでAzure VNetを作成する場合、コードに定義が必要な要素を事前に把握しておくことが重要です。 主な要素を整理します。

Resource Group（リソースグループ）

AzureのリソースはすべてResource Groupに属します。VNetを作る前に、Resource Groupが存在している必要があります。 Terraformでは、Resource GroupもVNetも同じコードで定義でき、作成順の依存関係は自動で解決されます。

Location（リージョン）

リソースをデプロイするAzureのリージョンです（例：Japan East）。 Resource GroupとVNetは同じLocationに配置するのが基本です。

VNet Address Space（アドレス空間）

VNet全体のIPアドレス範囲をCIDR形式で指定します。 オンプレミスや他のVNetと重複しないよう、事前にIPアドレス設計をしておく必要があります。

Subnet Address Prefix（サブネット）

VNet内をさらに分割した単位です。用途（アプリケーション、管理、Bastion、Private Endpointなど）ごとにサブネットを分けるのが一般的です。 オンプレミスのVLAN分割の感覚に近い部分があります。

NSG（Network Security Group）

サブネットまたはNICに関連付けるアクセス制御リストです。 オンプレミスのACLに近い役割を持ちますが、優先度（Priority）と方向（Inbound/Outbound）をセットで定義する点が異なります。 Terraformでは、NSGの定義とサブネットへの関連付けをそれぞれ別のresourceブロックで書きます。

Route Table（ルートテーブル）

サブネット内のトラフィックのルーティングを制御します。 既定ではAzureのシステムルートが適用されますが、NVAやVPN Gateway経由のルーティングを強制したい場合に追加します。 オンプレミスのスタティックルートの設定に近い感覚です。

Tags（タグ）

リソースに付与するメタデータです。環境（dev/prod）、担当者、プロジェクト名などを記録しておくと、 リソースの棚卸しやコスト管理に役立ちます。 Terraformでは全リソースのタグをコードで一元管理できます。

オンプレミスNWとの考え方の違い

ネットワークエンジニアとしてAzureのVNetを理解するとき、オンプレミスとの類比は助けになりますが、 完全に同じではない部分もあります。

  • 物理機器への設定投入ではない： Terraformで書くのはAzure上の論理リソースです。 スイッチやルーターのCLIにコマンドを投入するのではなく、クラウド上のAPIを通じてリソースを定義します。
  • VLANに近い感覚があるSubnetだが仕組みは異なる： SubnetはVNet内の論理分割ですが、オンプレミスのVLANとは違い、物理NICや802.1Qタギングとは無関係です。 Subnetはセキュリティ（NSG）や通信制御（Route Table）の適用単位として機能します。
  • ACLとNSGは似ているが優先度ベース： NSGはオンプレミスのACLに近い役割を持ちますが、許可・拒否のルールを優先度の数値（小さいほど優先）で管理します。 Denyルールが下位にある場合、Allowルールが先に適用されて通信が許可される点に注意が必要です。
  • VNet/Subnet/NSG/Route Tableはそれぞれ独立したリソース： Terraformでは、これらの関連付けもコードで明示的に定義します。 NSGをSubnetに関連付けるには別途 azurerm_subnet_network_security_group_association というリソースが必要です。

Terraformで書く前に設計しておくべきこと

コードを書き始める前に、以下の点を整理しておくと、修正の手戻りが減ります。

IPアドレス設計

VNetのアドレス空間と各サブネットのCIDRを事前に決めておきます。 後から変更するとリソースの再作成が必要になる場合があるため、最初にまとめて設計するのが望ましいです。

サブネット分割

用途ごとにサブネットを分けておくと、NSGやRoute Tableの適用範囲が明確になります。 例えば、Bastion専用サブネット（AzureBastionSubnet）は名前が固定されているなど、Azureのサービスによって制約がある場合もあります。

通信制御の方針

どのサブネット間の通信を許可・拒否するか、インターネットアクセスの要否、オンプレミスとの接続が必要かどうかを整理しておきます。 これがNSGのルール設計に直結します。

命名規則

Resource Group、VNet、Subnet、NSGなどのリソース名を統一しておきます。 後から命名を変えるとTerraformでリソースの再作成が発生する場合があるため、最初に決めておくことが重要です。

タグ設計

環境（dev/staging/prod）やプロジェクト名などのタグキーを事前に決めておくと、コード全体で統一したタグを付与できます。 Terraformでは変数や locals を使ってタグをまとめて管理できます。

将来的な拡張性

後からPrivate EndpointやVPN Gatewayを追加する可能性がある場合、それらに必要なサブネットのCIDRをあらかじめ確保しておくと、 後の設計変更を最小限にできます。

plan / apply の重要性

Terraformを使う大きなメリットの一つが、terraform plan によって変更内容を事前に確認できることです。

terraform plan を実行すると、作成・変更・削除されるリソースの一覧が表示されます。 手作業でポータルから変更する場合と比べて、変更の意図と実際の差分を突き合わせやすくなります。 ネットワーク変更で言えば、コマンド投入前の「変更前後の比較確認」に相当するステップです。

terraform apply は plan の内容を実際に適用します。 確認プロンプトで yes を入力するまで変更は走らないため、誤操作のリスクを減らせます。

特に注意が必要なのは、変更内容によってはリソースの再作成（削除→新規作成）が発生する場合がある点です。 plan の出力で -/+ と表示されるものはその対象です。 IPアドレスや名前など変更不可な属性を変えた場合に発生しやすいため、plan の確認を習慣化することが重要です。

今後やりたいこと

前提知識の整理ができたので、今後は実際に手を動かして検証を進める予定です。 ステップとして以下を想定しています。

  • TerraformでResource Groupを作成する
  • TerraformでVNetとSubnetを作成する

  • Route Tableを定義して通信経路を制御する
  • Azure BastionやPrivate Endpoint構成へ発展させる

まとめ

TerraformでAzure VNetを作成する前に、Resource Group、Location、Address Space、Subnet、NSG、Route Table、Tagsという 構成要素を整理しておくことが重要です。

オンプレミスのネットワーク設計と考え方が近い部分はありますが、物理機器への設定投入とは異なるアプローチのため、 クラウド特有のリソースの定義方法や関連付けの仕組みを理解してからコードを書き始めるのがよいと感じました。

plan / apply の流れを活用しながら、実際に手を動かして検証を進めていきます。
