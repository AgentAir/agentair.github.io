Azure Arcは再インストール前にまず状態確認：UbuntuがConnectedだった話

2026-05-13 / Azure / Azure Arc / Sentinel

Home Articles About

はじめに

Windows 11とUbuntuをAzure Arcで管理し、最終的にはMicrosoft Sentinel / Log Analyticsに接続して ログ収集できる構成を確認していました。

先にWindows 11側のAzure Arc再接続が完了したため、次はUbuntu側も同じように Azure Arcを再インストールして再接続し直すつもりでいました。 以前のリソースや接続情報が残っているかもしれないので、いったん整理した方が確実だと思っていたためです。

ただ、実際に確認してみるとUbuntuはすでにAzure ArcでConnected状態でした。 そのため、今回の結論は「再インストールする」ではなく、 「まず現在の接続状態を確認し、問題がなければそのまま使う」になりました。

最初に考えていたこと

最初の理解では、Windows 11と同じようにUbuntu側もAzure Arcを再インストールし直す必要があると考えていました。 Windows側で再接続作業を行った流れがあったため、Ubuntu側も同じ手順でそろえた方がよいと思っていたからです。

特に気になっていたのは、Azure Portal上に以前のArcマシンリソースが残っていたり、 ローカル側に古い接続情報が残っていたりする可能性でした。 Sentinel / Log Analyticsへ進む前に、Arcの状態をきれいにしておきたいという気持ちがありました。

ただ、この時点では「きれいにする」ことと「再インストールする」ことを少し同じものとして考えていました。 後から振り返ると、ここが今回の学びの入口でした。

実際に確認したこと

Ubuntu側で状態を確認したところ、Azure Arcの接続状態はすでにConnectedでした。 つまり、Azure Arcエージェントは動作していて、Azure側のArcマシンリソースとして認識されている状態です。

この状態であれば、再インストールは不要です。 むしろ、何も考えずに削除・再接続を行うと、Azure Portal上に別のArcマシンリソースが作成されたり、 どのリソースが現在のUbuntuを指しているのか分かりにくくなったりする可能性があります。

そのため、接続済みのUbuntuに対して最初にやるべきことは再インストールではなく、 azcmagent showで現在の接続情報を確認することだと整理しました。

確認コマンド

Ubuntu側では、次のコマンドでAzure Connected Machine Agentの状態を確認します。 

sudo azcmagent show

ここで見るポイントは、単にConnectedかどうかだけではありません。 Azure Portal上のAzure Arcマシン一覧に表示されている内容と、Ubuntu側の表示内容が一致しているかを確認します。

確認項目 見る理由
Status / Agent Status Connectedになっているか確認する
Resource Name Azure Portal上のArcマシン名と一致するか確認する
Resource Group 想定したリソースグループに登録されているか確認する
Subscription ID 誤ったサブスクリプションに登録されていないか確認する
Tenant ID 想定したテナントに接続されているか確認する
Location japaneastなど想定リージョンか確認する

今回は、UbuntuがすでにConnectedであることを確認できたため、 再インストールではなく、これらの接続情報の照合に進めばよいという判断になりました。

再インストールが必要なケース

Azure Arcの再インストールや再接続は、最初にやる作業ではなく、 状態確認後に必要な場合だけ行う作業だと理解しました。 例えば、次のようなケースでは再インストールや再接続を検討します。

  • azcmagent showでConnectedではない
  • Azure Portal側にArcマシンが存在しない
  • 異なるSubscription / Resource Group / Tenantに接続されている
  • Azure側のArcリソースを削除したのに、Ubuntu側に古い接続情報だけ残っている
  • エージェント自体が壊れている、またはazcmagentコマンドが存在しない

逆に言えば、Azure Portal側のArcマシンとUbuntu側のazcmagent showの情報が一致し、 状態もConnectedであれば、再インストールする理由はありません。

Sentinel / Log Analytics連携へ進む流れ

Azure Arcの接続確認ができたら、次はログ収集の設定に進みます。 自分の中では、次の順番で確認していく流れとして整理しました。 

Ubuntu の Azure Arc 接続確認
  ↓
Azure Portal の Azure Arc マシン一覧で確認
  ↓
Log Analytics Workspace を確認
  ↓
Data Collection Rule / DCR を作成
  ↓
Azure Monitor Agent / AMA を Arc マシンへ適用
  ↓
Log Analytics にログが入るか確認
  ↓
Microsoft Sentinel で確認

ここで重要なのは、Sentinel側からいきなり考え始めないことです。 SentinelはLog Analytics Workspaceに入ってきたログを分析・検知に使う側なので、 まずはUbuntuがArcマシンとして正しく認識され、AMAとDCRでログ収集できる状態を作る必要があります。

今回の学び

今回の一番大きな学びは、Azure Arcは「再インストールすればよい」ではなく、 まず現在の状態を確認するものだということです。

  • Azure Arcは、再インストール前に現在の接続状態を確認する
  • Connectedなら、再インストールしない方が安全
  • 再接続を繰り返すと、Azure Portal上に別リソースが作成される可能性がある
  • Arc管理では、ローカルのazcmagent showとAzure Portal上のリソース情報の一致確認が重要
  • Sentinel連携では、Arc接続確認後にAMA / DCRの設定へ進む

Windows 11側で再接続を行った流れがあったため、Ubuntu側も同じようにやり直すものだと思っていました。 しかし、UbuntuがすでにConnectedであれば、やるべきことは再インストールではなく、 接続先のSubscription、Resource Group、Tenant、Location、Resource Nameを確認することでした。

次にやること

次は、UbuntuがAzure Arcマシンとして正しく見えていることを前提に、 Log Analytics Workspace、Data Collection Rule（DCR）、Azure Monitor Agent（AMA）の設定を確認します。

そのうえで、Log Analyticsにログが入るかを確認し、最後にMicrosoft Sentinel側でログを検索できるかを見ていきます。 Arcの再インストールから始めるのではなく、Arc接続の確認からログ収集設定へ進む、という流れで検証を続けます。
