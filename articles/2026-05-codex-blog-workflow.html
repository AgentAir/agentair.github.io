CodexでGitHub Pagesブログを安全に更新する実践手順

2026-05-03 / Codex / GitHub Pages / Blog Operation

Home Articles About

はじめに

Codexを使うと、GitHub Pagesで公開している静的HTMLブログの更新をかなり速く進められます。 ただし、公開リポジトリを直接扱う以上、最初から大きな編集を任せるのではなく、小さく確認しながら進めることが重要です。

この記事では、GitHub Pagesブログをローカルにcloneしてから、Codexに構成確認や記事追加を依頼し、差分を見ながら安全に更新する手順を整理します。

背景

対象は、GitHub Pagesで公開している手書きHTML/CSSのブログです。 最初にリポジトリをローカル環境へcloneし、作業前に git status で作業ツリーの状態を確認しました。

その時点では、Codexにいきなりファイル編集を任せず、まず既存のHTML構造、記事一覧、トップページ、記事テンプレートの関係だけを確認させました。 どのファイルを触る必要があるかを先に把握してから、実際の編集に進むためです。

結論（先に）

Codexでブログを安全に更新する基本は、作業範囲を小さく決め、編集前後に差分を確認し、commit単位を小さく保つことです。

  • 作業前に git status で作業ツリーを確認する
  • Codexには最初から編集させず、まず構成確認だけを依頼する
  • 編集対象ファイルを明示し、CSSや既存構造を不用意に変更しない
  • git diff と git diff --cached で差分を確認する
  • 公開禁止情報を入れず、commit単位を小さくする

実際の手順

今回の作業では、まずGitHub Pagesブログのリポジトリをローカルにcloneし、現在の状態を確認しました。 作業ツリーが想定外に汚れていると、Codexが加えた変更と既存の変更を切り分けにくくなります。 

git status

次に、Codexへ既存構成の確認だけを依頼しました。 この段階では、記事の追加やCSS変更は行わず、トップページ、記事一覧、個別記事、テンプレートの役割を把握することに絞りました。

その後、運用ルールを README.md に追加しました。 Codexへ任せてよい作業、任せない作業、公開禁止情報、commit前の確認項目を明文化しておくと、次回以降の依頼も再現しやすくなります。

記事追加は、いきなり大きく増やすのではなく、小さく試しました。 既存のHTML構造に合わせて個別記事を1本追加し、articles.html と index.html に必要なリンクだけを追加します。

さらに、記事作成を繰り返しやすくするために articles/template.html を作成し、Codexへ記事作成を依頼するための docs/codex-article-prompt.md も作成しました。 ただし、仕組みを大きく変えるのではなく、既存の静的HTML運用の範囲に収めました。 

git diff
git add articles/2026-05-codex-blog-workflow.html articles.html index.html
git diff --cached

重要ポイント

Codexを使うときは、依頼内容を具体的にするほど安全に進めやすくなります。 特に公開ブログでは、編集してよいファイルと編集してはいけないファイルを明示することが重要です。

  • CSS変更なしで既存HTML構造を維持する
  • 記事追加、一覧追加、トップページ追加を分けて確認する
  • 差分確認では、意図しないファイルが含まれていないかを見る
  • commitは「記事1本追加」「運用ルール追加」のように小さくする
  • 会社情報、顧客情報、実IP、認証情報、SAS URL、試験問題本文は入れない

Codexは文章やHTMLを素早く作れますが、公開してよい内容かどうかの最終判断は人間が行う必要があります。 特に検証ログやエラー例を書く場合は、固有名詞や値を一般化してから公開します。

よくある失敗

よくある失敗は、便利だからといって最初から大規模改修を依頼してしまうことです。 記事追加のつもりが、CSS、ナビゲーション、既存記事の文面まで変わると、レビューが難しくなります。

  • 作業前の git status を確認しない
  • 編集してよいファイルを指定しない
  • git diff を見ずにcommitする
  • 複数の目的を1つのcommitに混ぜる
  • 検証ログに公開禁止情報を残したまま記事化する

失敗を避けるには、Codexに「まず確認だけ」「次に1ファイルだけ」「最後に一覧へリンク追加」のように段階を分けて依頼するのが有効です。

まとめ

CodexでGitHub Pagesブログを更新するときは、小さく始めて、差分を見て、必要な分だけcommitする流れが基本です。 ローカルにcloneし、git status で状態を確認し、構成確認から始めることで、意図しない変更を減らせます。

articles/template.html と docs/codex-article-prompt.md のような最小限の型を用意しておくと、次の記事追加も再現しやすくなります。 ただし、公開禁止情報を入れないこと、大規模改修にしないこと、差分確認を省略しないことは毎回守るべきです。
