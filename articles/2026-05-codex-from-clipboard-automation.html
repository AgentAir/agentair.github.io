Codex依頼をクリップボードから保存して記事作成につなげる流れ

2026-05-14 / Codex / GitHub Pages / Blog Operation

Home Articles About

はじめに

今回のCodex依頼は、ブログ記事の本文を直接渡すものではなく、ローカルのPowerShellスクリプトを実行するだけの短い内容でした。 最初は「この2行だけで何の記事にすればよいのか」と迷いましたが、実際にはCodexへの記事作成依頼をクリップボードから保存し、作業ブランチ上でCodexを起動するための入口でした。

この記事では、Start-CodexFromClipboard.ps1 が担っている役割と、そこから分かった半自動の記事作成フローを整理します。 具体的な個人情報、ID、IPアドレス、トークン、SAS URL、秘密鍵などは記載しません。

背景 / 目的

手書きHTMLのGitHub Pagesブログでは、新しい記事を追加するときに、記事HTMLの作成、articles.html へのリンク追加、差分確認を毎回行います。 その作業自体は単純ですが、依頼文の保存、ログ取得、作業前の安全確認まで含めると、手順が増えて抜け漏れが起きやすくなります。

そこで、ChatGPT側で作ったCodex依頼文をクリップボードにコピーし、PowerShellスクリプトで保存してからCodexを実行する形にしました。 目的は、記事作成を完全自動化することではなく、作業開始前の確認と依頼文の保管を標準化することです。

最初の誤解

最初に見えた依頼は、次の2行だけでした。 

cd "$env:USERPROFILE\Documents\GitHub\agentair.github.io"

.\scripts\Start-CodexFromClipboard.ps1

この時点では、単にCodexを起動するだけのメモなのか、記事化するべき運用手順なのかが分かりませんでした。 しかし、保存された依頼ファイルとスクリプトを確認すると、これは「クリップボードにあるCodex依頼を保存し、安全確認をしてからCodexに渡す」ための運用でした。

つまり、主役は2行のコマンドそのものではなく、その裏側にある記事作成ワークフローでした。

確認したこと

スクリプトの中身を読むと、次のような流れになっていました。

  1. リポジトリ、依頼保存用フォルダ、ログ保存用フォルダの存在を確認する
  2. git、codex、必要に応じて code コマンドの有無を確認する
  3. クリップボードからCodex依頼文を読み取り、空欄や明らかに無関係な内容でないか確認する
  4. 依頼文を codex-inbox 配下へUTF-8で保存する
  5. git status で作業ツリーが汚れていないか確認する
  6. main を更新し、日付ベースの作業ブランチを作る
  7. 記事作成ルールを含めた依頼文を組み立て、codex exec に渡す
  8. 作業後に git status、git diff --stat、簡易的な秘密情報チェックを実行する

議論中に出た問い

今回の依頼から読み取れた問いは、技術的には次の3つでした。

  • Codexに渡した依頼文を、あとから追跡できる形で残せるか
  • 作業前に、別の未コミット変更が混ざっていないことを確認できるか
  • Codexの出力をそのまま公開せず、人間がレビューする前提にできるか

TODO: 実際の会話で出た細かい質問や判断理由は、保存された依頼ファイルだけでは確認できません。 必要であれば、人間側で当時の会話ログを見直して追記します。

最終的な理解

このスクリプトの結論は、Codexを「いきなり編集する道具」として使うのではなく、依頼文の保存、作業前チェック、ブランチ作成、実行後の確認まで含めた記事作成フローに組み込むことです。

特に重要なのは、スクリプトが作業ツリーの未コミット変更を検知したら止まる点です。 これにより、Codexが加えた変更と、もともと存在していた変更が混ざりにくくなります。

また、実行後に簡易的な秘密情報チェックを行うことで、公開ブログに載せてはいけない文字列が混ざっていないかを確認する入口にもなります。 ただし、これはあくまで簡易チェックなので、最終的な公開判断は人間が行う必要があります。

学んだこと

今回の学びは、半自動化では「実行を速くすること」よりも「毎回同じ安全確認を通すこと」が重要だという点です。 Codexに記事作成を任せる場合でも、依頼文、ログ、差分、レビュー観点が残っていれば、あとから作業内容を追いやすくなります。

  • クリップボード経由でも、依頼文をファイルとして残せば再確認しやすい
  • 作業前の git status は、半自動化するほど重要になる
  • 秘密情報チェックは便利だが、公開前の人間レビューの代わりにはならない

変更ファイルとレビュー観点

今回の記事化で変更したファイルは次の通りです。

  • articles/2026-05-codex-from-clipboard-automation.html: 新規記事を追加
  • articles.html: GitHub / Blog Operation セクションへ記事リンクを追加

人間がレビューするポイントは、記事本文が実際の運用意図と合っているか、TODOとして残した会話経緯を追記すべきか、公開してはいけない情報が含まれていないかです。

まとめ

Start-CodexFromClipboard.ps1 は、Codex依頼をクリップボードから読み取り、保存し、作業前後の確認を通して記事作成につなげるための入口です。 最初は単なる起動コマンドに見えましたが、実際には安全に半自動化するための運用手順でした。

今後は、依頼文の保存、作業ツリー確認、差分確認、秘密情報チェック、人間レビューを1つの流れとして扱うことで、GitHub Pagesブログの記事追加を安定して続けやすくなります。
