Azure VM以外でMicrosoft Sentinelを使うにはAzure Arcが必須なのか整理した

2026-05-13 / Azure / Microsoft Sentinel / Azure Arc / AMA

Home Articles About

はじめに

Microsoft SentinelとAzure Arc、Azure Monitor Agent（AMA）の関係を調べている中で、 最初は「Azure VM以外のシステムでSentinelを使うには、Azure Arcが必須」と理解していました。

ただ、調べていくうちに少し違和感が出てきました。 Sentinelはログを分析するためのSIEM/SOARサービスであり、ログを入れる方法はAMAだけではありません。 そう考えると、「Azure VM以外なら必ずArcが必要」と言い切るのは少し雑ではないか、という疑問が出てきました。

今回はその理解を整理し直したメモです。 結論から言うと、ポイントは「どの方式でLog Analytics Workspaceへログを入れるか」です。

最初に混乱したこと

最初の理解では、Microsoft Sentinel、Log Analytics Workspace、AMA、DCR、Azure Arcを まとめて「Sentinelにログを送るための一式」として見ていました。 そのため、Azure外のサーバーを扱うならArcが必須、と単純に覚えそうになっていました。

しかし、実際にはそれぞれの役割は分かれています。 Sentinelはログの分析・検知・インシデント管理を行う側であり、 その前段にあるログ収集の方式によって、Arcが必要かどうかが変わります。

各コンポーネントの関係

自分の中では、まず以下のように役割を分けると理解しやすくなりました。

  • Microsoft Sentinel： Log Analytics Workspaceに入ったログを使って、分析、検知、インシデント管理を行うSIEM/SOARサービス。
  • Log Analytics Workspace： Sentinelが利用するログの保存先。Sentinelはここに入ったログを検索・分析に使う。
  • AMA（Azure Monitor Agent）： Azure Monitor Agentのこと。サーバーのイベントログ、Syslog、パフォーマンス情報などを収集し、 Log Analytics WorkspaceやSentinelで使えるように送信するエージェント。
  • DCR（Data Collection Rule）： Data Collection Ruleのこと。AMAが何のログを収集するか、どこへ送るかを定義するルール。
  • Azure Arc： オンプレミス、VMware、他クラウドなどAzure外のサーバーをAzure上の管理対象リソースとして扱えるようにする仕組み。

つまり、AMA + DCRでサーバーのログを収集する場合、 DCRをどのマシンに関連付けるのか、という管理対象が必要になります。 Azure VMであれば最初からAzureリソースですが、Azure外のサーバーはそのままではAzure上の管理対象リソースではありません。 そこでAzure Arcが関係してきます。

Azure VMの場合

Azure VMは、作成された時点でAzure Resource Manager上のリソースです。 そのため、Azure VM拡張機能としてAMAを導入でき、DCRもAzure VMに関連付けられます。 この場合、Azure Arcは不要です。 

Azure VM
  → Azure VM拡張機能としてAMAを導入
  → DCRを関連付け
  → Log Analytics Workspace
  → Microsoft Sentinel
  → Azure Arcは不要

ここでArcを使わない理由は、Azure VMがすでにAzureの管理対象だからです。 ArcはAzure外のリソースをAzureに接続するための仕組みなので、 すでにAzureリソースであるVMに対して、通常はArcを追加する必要はありません。

オンプレ / VMware / 他クラウドのサーバーの場合

一方で、オンプレミス、VMware、AWS、GCPなどにあるサーバーは、 そのままではAzure上のVMリソースではありません。 そのサーバーをAMA + DCRの対象としてAzure側から管理したい場合は、 Azure ArcでAzureに登録し、Azure Arc対応サーバーとしてリソース化する必要があります。 

オンプレ / VMware / 他クラウドのサーバー
  → Azure ArcでAzureに登録
  → Azure Arc対応サーバーとしてAzure上にリソース化
  → AMAを導入
  → DCRを関連付け
  → Log Analytics Workspace
  → Microsoft Sentinel

ここで必要なのは、「Sentinelを使うためのArc」ではなく、 「Azure外のサーバーをAzure管理リソースとして扱い、AMA + DCRを適用するためのArc」です。 この違いを分けて考えると、だいぶ整理しやすくなりました。

Arcが不要なログ連携方式もある

重要なのは、Sentinelへのログ連携がすべてAMA + DCRだけではないことです。 Sentinelにはさまざまなデータコネクタや取り込み方式があります。 そのため、次のようなケースでは、ログ発生元のシステムにAzure Arcを入れる必要がないことがあります。

  • Azure VMからAMAでログを収集する場合
  • SaaS / Microsoft 365 / Defender系コネクタで連携する場合
  • ネットワーク機器がSyslog / CEFをログフォワーダへ送る構成
  • LogstashやLogs Ingestion APIを使って取り込む構成
  • APIで直接Log Analytics Workspaceに送信する構成

ただし、Syslog / CEFについては少し注意が必要です。 ネットワーク機器やセキュリティ機器そのものはArc対応サーバーにするわけではありません。 一方で、非AzureのLinuxサーバーをログフォワーダとして用意し、そのフォワーダにAMAを導入する場合は、 そのフォワーダ側をAzure Arc対応サーバーとして扱う構成になりやすいです。

Arcが必要になりやすいケース

逆に、Azure外のサーバーそのものをAMA + DCRで直接収集対象として扱う場合は、 Azure Arcが必要になりやすいと整理しました。

  • オンプレWindows ServerのイベントログをAMA + DCRで収集する場合
  • VMware上のLinux / Windows ServerをAMA + DCRで収集する場合
  • AWS / GCP上のVMをAzure Monitor Agentで管理する場合
  • 非AzureのLinuxサーバーをSyslog / CEFログフォワーダとして使い、そのフォワーダにAMAを導入する場合

この場合、Azure外のサーバーに対してDCRを関連付ける必要があります。 その前提として、そのサーバーがAzure上でArc対応サーバーとして見えている必要がある、という理解です。

条件分岐で整理する

今回の学びを、実務メモとして使いやすい形にすると次のようになります。

ログ連携の考え方 Azure Arcの要否 理由
Azure VMにAMAを入れてDCRで収集する 不要 Azure VMは最初からAzure管理リソースであり、VM拡張機能としてAMAを導入できるため。
オンプレ / VMware / 他クラウドのサーバーにAMAを入れてDCRで収集する 必要になりやすい Azure外のサーバーをArc対応サーバーとして登録し、DCRの関連付け対象にする必要があるため。
SaaS / Microsoft 365 / Defender系コネクタを使う 不要なことが多い サーバーにAMAを入れる方式ではなく、サービス側のコネクタで連携するため。
ネットワーク機器がSyslog / CEFをフォワーダへ送る 機器自体は不要 機器はSyslog / CEFを送るだけ。フォワーダにAMAを入れる場所が非Azureなら、フォワーダ側でArcが関係する。
Logs Ingestion APIやAPI連携で送信する 不要なことが多い Azure管理リソースとしてサーバーにDCRを関連付ける方式ではなく、APIでログを投入するため。

理解が変わったポイント

最初は「Azure VM以外 = Azure Arc必須」と覚えそうになっていました。 しかし、調べ直してみると、正確にはもう少し条件付きでした。

正確には、「非AzureサーバーをAMA + DCR方式でSentinelに直接収集対象として扱う場合はAzure Arcが必要」と考える方が近いです。 一方で、SaaSコネクタ、Microsoft 365 / Defender系コネクタ、Syslog / CEF転送、Logstash、Logs Ingestion API、 直接API送信など、Arcを前提にしないログ連携方式もあります。

つまり、Sentinelを使うかどうかではなく、ログ収集の方式を見る必要があります。 ここを分けて考えることで、「Arcが必要な構成」と「Arcなしでよい構成」を判断しやすくなりました。

まとめ

Microsoft Sentinel を使うこと自体に Azure Arc が必須なわけではない。 Azure Arc が必要になるのは、主に Azure 外のサーバーを Azure の管理対象リソースとして登録し、AMA + DCR によってログ収集する場合である。

今回の整理で、自分の中では「Sentinelの話」と「ログ収集方式の話」を分けて考えるのが大事だと分かりました。 Sentinelにログを入れる方法は複数あり、その中でAzure外のサーバーをAMA + DCRの管理対象にする場合にAzure Arcが出てくる、 という理解で進めるのが実務上も分かりやすそうです。
