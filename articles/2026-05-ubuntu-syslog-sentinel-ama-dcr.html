Ubuntu の Syslog を Azure Arc / AMA / DCR 経由で Microsoft Sentinel に収集する手順

2026-05-14 / Azure / Microsoft Sentinel / Azure Arc / AMA / DCR

Home Articles About

はじめに

今回は、Ubuntu の Syslog を Microsoft Sentinel に収集できるようにした検証を整理します。 最初は、Ubuntu が Azure Arc に接続されていれば、そのまま Sentinel にログが送られると思っていました。 しかし実際には、Azure Arc に接続しただけでは不十分でした。

Sentinel に Ubuntu の Syslog を入れるには、Arc、AMA、DCR、Log Analytics Workspace の役割を分けて見る必要があります。 今回の記事では、単なる手順だけではなく、どこで勘違いし、何を確認して理解が変わったのかを残します。

最初の理解

最初の理解は、かなり単純でした。 Ubuntu を Azure Arc に接続すれば、Azure 側から管理できるようになり、そのまま Microsoft Sentinel にログも送られるだろうと考えていました。

しかし、実際に確認していくと、Arc 接続はあくまで「Azure から対象サーバーを管理対象として見えるようにする」ための入口でした。 Syslog を収集して Log Analytics Workspace に送るには、次の条件がそろっている必要があります。

  • Azure Arc 対応サーバーとして Azure に登録されていること
  • Azure Monitor Agent / AMA が導入されていること
  • Data Collection Rule / DCR が対象の Ubuntu Arc マシンに関連付いていること
  • DCR の Collect 設定で Syslog の Facility / Severity が設定されていること

今回の構成

今回は、Ubuntu 上で発生させた Syslog を rsyslog 経由で Azure Monitor Agent に渡し、DCR の設定に従って Log Analytics Workspace へ送信しました。 Microsoft Sentinel はその Workspace 上のログを使うため、最終的には Sentinel の画面から Syslog テーブルを確認します。 

Ubuntu
  ↓ Syslog / logger
rsyslog
  ↓
Azure Monitor Agent / AMA
  ↓ DCR
Log Analytics Workspace
  ↓
Microsoft Sentinel
  ↓
Syslog テーブルで確認

ここで大事なのは、Sentinel が直接 Ubuntu へ取りに行くわけではないという点です。 Ubuntu 側で AMA が動き、DCR によって「どのログを集めるか」が配布され、その結果が Log Analytics Workspace に入ります。

役割の整理

初心者のうちは、Arc、AMA、DCR、Sentinel がまとめて「ログ連携の仕組み」に見えます。 しかし、詰まったときに切り分けるには、役割を分けて考える必要があります。

  • Azure Arc: Ubuntu を Azure 上の管理対象リソースとして登録する仕組み。
  • Azure Monitor Agent / AMA: Ubuntu 上で動作し、ログを収集して Azure Monitor / Log Analytics Workspace に送るエージェント。
  • Data Collection Rule / DCR: AMA が何を収集し、どこへ送るかを定義するルール。
  • Log Analytics Workspace: Sentinel が参照するログの保存先。
  • Microsoft Sentinel: Workspace に入ったログを使って検索、分析、検知、調査を行う SIEM/SOAR サービス。

実施した内容

今回は、次の順番で確認しました。 ポイントは、Arc 接続、AMA の疎通、DCR の配布、Syslog の到達を別々に確認したことです。

  1. Ubuntu が Azure Arc に接続されていることを確認する。
  2. Sentinel の Data connectors から Syslog via AMA を選択する。
  3. DCR を作成する。
  4. DCR の Resources で Ubuntu の Azure Arc 対応サーバーを選択する。
  5. Ubuntu 側で configchunks に DCR 設定が配布されたことを確認する。
  6. Heartbeat テーブルで AMA の接続を確認する。
  7. logger コマンドで Syslog テストログを生成する。
  8. Syslog テーブルで sentinel-test が見えることを確認する。

確認コマンド

Ubuntu 側では、まず Arc と AMA の状態を確認しました。 

sudo azcmagent show

systemctl status azuremonitoragent --no-pager

次に、AMA に DCR の設定が配布されているかを確認しました。 configchunks は、Azure 側で関連付けた DCR の設定が AMA 側に届いているかを見るために確認します。 ここに何も増えない場合、AMA は動いていても「何を収集するか」の設定が届いていない可能性があります。 

sudo ls -l /etc/opt/microsoft/azuremonitoragent/config-cache/configchunks/

sudo grep -R -i "syslog" /etc/opt/microsoft/azuremonitoragent/config-cache/configchunks/

Syslog 側では、AMA 関連の rsyslog 設定が入っているかも確認しました。 

sudo ls -l /etc/rsyslog.d/

sudo ls -l /etc/rsyslog.d/*azuremonitoragent*

最後に、logger コマンドでテスト用の Syslog を発生させました。 Facility と Severity を分けて投げることで、DCR 側の Collect 設定と合っているかを確認しやすくしています。 

logger -p authpriv.err "sentinel-test ubuntu authpriv err $(hostname) $(date -Is)"
logger -p daemon.warning "sentinel-test ubuntu daemon warning $(hostname) $(date -Is)"
logger -p user.info "sentinel-test ubuntu user info $(hostname) $(date -Is)"

KQL で確認する

まずは Heartbeat テーブルで、AMA が Log Analytics Workspace に接続できていることを確認しました。 初回確認では範囲を狭くしすぎると見落とすことがあるため、ago(24h) のように広めに見ました。 

Heartbeat
| where TimeGenerated > ago(24h)
| where Category == "Azure Monitor Agent"
| project TimeGenerated, Computer, OSType, Category
| order by TimeGenerated desc

次に、logger で生成した sentinel-test が Syslog テーブルに入っているかを確認しました。 

Syslog
| where TimeGenerated > ago(24h)
| where SyslogMessage contains "sentinel-test"
| project TimeGenerated, Computer, HostName, Facility, SeverityLevel, ProcessName, SyslogMessage
| order by TimeGenerated desc

収集されている Facility と Severity の内訳を見るには、次のように集計しました。 

Syslog
| where TimeGenerated > ago(24h)
| summarize count() by Computer, HostName, Facility, SeverityLevel
| order by count_ desc

途中で詰まったところ

今回いちばん詰まったのは、最初に次のコマンドを実行しても、DCR 設定ファイルが配布されていなかったことです。 

sudo ls -l /etc/opt/microsoft/azuremonitoragent/config-cache/configchunks/

この時点では、Ubuntu は Arc に接続されており、AMA も動いているように見えていました。 そのため、最初は「なぜ Syslog が来ないのか」が分かりにくい状態でした。

原因は、DCR の Resources に Ubuntu の Azure Arc 対応サーバーを正しく選択していなかったことでした。 DCR を作成していても、対象の Arc マシンに関連付いていなければ、その Ubuntu には収集設定が配布されません。

その後、DCR の Resources で Azure Arc の Ubuntu マシンを選択したところ、configchunks 配下にファイルが増えました。 これにより、DCR が Ubuntu に配布されたことが確認できました。 ここで初めて「AMA は動いている」と「DCR が対象に配布されている」は別の確認だと理解できました。

Heartbeat は見えるのに Syslog が見えない理由

Heartbeat が見える場合、AMA 自体は動作しており、Log Analytics Workspace へ接続できている可能性が高いです。 しかし、それだけでは Syslog 収集が成功しているとは言えません。

Syslog が見えない場合は、少なくとも次の観点を分けて確認する必要があります。

  • DCR が対象の Ubuntu Arc マシンに関連付いているか。
  • configchunks に DCR の設定が配布されているか。
  • DCR の Collect 設定で、投げた Facility / Severity が収集対象になっているか。
  • rsyslog 側に AMA 関連の設定が入っているか。
  • Syslog の反映に数分かかっているだけではないか。

今回は、DCR の Resources 関連付けが抜けていたため、Heartbeat は見えるが Syslog が見えない状態になっていました。

最終的に理解したこと

今回の検証で、最終的には次の理解に落ち着きました。

  • Heartbeat が見える場合、AMA 自体は動作している。
  • configchunks にファイルがある場合、DCR は Ubuntu に配布されている。
  • Syslog テーブルに sentinel-test が出れば、Ubuntu の Syslog 収集は成功。
  • Syslog はすぐ表示されない場合があるため、数分待つ必要がある。
  • 初回確認では ago(1h) より ago(24h) のように広めに見る方がよい。
  • Azure Arc 接続だけでは Sentinel にログは送られず、AMA と DCR の関連付けが重要。

まとめ

Ubuntu の Syslog を Microsoft Sentinel に収集するには、Azure Arc に接続するだけでは足りません。 Arc は Ubuntu を Azure 側の管理対象として見えるようにする入口であり、実際にログを送るのは AMA、何を収集するかを決めるのは DCR です。

今回の詰まりポイントは、DCR の Resources に Ubuntu の Azure Arc 対応サーバーを正しく選択していなかったことでした。 configchunks を見ることで、DCR が Ubuntu に配布されたかを確認でき、Heartbeat と Syslog を分けて見ることで、AMA の接続確認と Syslog 収集確認を切り分けられました。

最終確認は、logger で発生させた sentinel-test が Syslog テーブルに表示されることです。 ここまで確認できれば、Ubuntu の Syslog を Azure Arc / AMA / DCR 経由で Microsoft Sentinel に収集できたと言えます。
