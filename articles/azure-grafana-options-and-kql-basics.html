AzureのGrafana選択肢とKQLの基本を整理する

2026-05-20 / Azure / Grafana / Azure Monitor / Log Analytics / KQL / Sentinel

Home Articles About

はじめに

前回、Azure Monitor dashboards with Grafana を使って、Microsoft Sentinel / Log Analytics の Heartbeat を Grafana 形式で表示できることを確認しました。 その中で気になったのが、Azure Monitor dashboards with Grafana と Azure Managed Grafana の違いです。

最初はどちらも「Azure の Grafana」なので、同じものを別の入口から見ているだけなのかと誤解しそうになりました。 さらに Grafana で使っている KQL についても、SQL と同じようなものなのか、どう理解すればよいのかを整理したくなりました。

今回は、Grafana の選択肢と KQL の基本を、自分の理解に合わせて整理します。

関連: SentinelのログをAzure Monitor dashboards with Grafanaで可視化してみた

Azure Monitor dashboards with Grafana は何か

Azure Monitor dashboards with Grafana は、Azure Portal 内で使える Grafana 形式のダッシュボード機能です。 ユーザーは Grafana サーバーを構築せず、Grafana 本体の運用も行いません。 その意味では、SaaS 的に使える Grafana ビューと考えると分かりやすいです。

ただし、Grafana Cloud のような汎用 SaaS Grafana ではありません。 Azure Portal 内で、Azure Monitor 系の監視データを Grafana 形式で見るための機能と理解するとよさそうです。

今回の理解では、Azure Monitor dashboards with Grafana は、Azure Portal に組み込まれた SaaS 的な Grafana ビューです。 外部データソースを自由につなぐ汎用 Grafana というより、Azure Monitor / Log Analytics / Azure Resource Graph / Azure Data Explorer など、 Azure 側にある監視・分析データを可視化する用途に向いています。

Azure Managed Grafana は何か

Azure Managed Grafana は、Azure 上に作成するマネージド Grafana サービスです。 Azure App Service や Azure Database のように、基盤運用の多くを Azure 側に任せる PaaS 的なサービスとして理解すると分かりやすいです。

ユーザーは OS に Grafana をインストールしたり、OS 更新をしたりする必要はありません。 Microsoft 側が Grafana サービスの基盤を管理してくれます。 ただし、完全に自由な自前 Grafana ではなく、Azure のマネージドサービスとして提供される Grafana を使う形です。

つまり、Azure Managed Grafana は素の Grafana をそのまま自分で入れるものではなく、 Azure 上で Microsoft が管理してくれる Grafana サービスです。

素のGrafanaを使う場合

素の Grafana を使いたい場合は、Azure VM、オンプレ VM、Docker、Kubernetes などに自分で Grafana をインストールします。 Azure で考えるなら、IaaS 上に自分で構築する形です。

自由度は高い一方で、OS、Grafana のアップデート、証明書、バックアップ、可用性、認証設定などを自分で管理する必要があります。 検証用途や自由に触りたい場合はよいですが、運用負荷も自分持ちになります。

外部装置は見られるのか

Azure Monitor dashboards with Grafana は、外部ストレージやネットワーク機器へ直接接続して監視するための汎用 Grafana ではありません。 外部装置へ直接 SNMP や API で取りに行く用途ではない、という整理です。

ただし、外部装置やオンプレサーバーのログを Azure Monitor / Log Analytics に取り込んでいれば表示できます。 つまり表示対象は、「Azure Monitor / Log Analytics など Azure 側に集約された監視データ」です。 直接収集したい場合は、Azure Managed Grafana や素の Grafana を検討するほうが自然です。

Managed Grafanaでできること

Azure Managed Grafana は、ダッシュボード表示だけでなく、本格的な Grafana 運用機能を使うためのサービスです。 Grafana アラート、スケジュールレポート、外部データソース連携、認証・権限管理、Managed Identity、サービスプリンシパル、 Private Link などの閉域構成を使う場合に検討対象になります。

ただし、「スケジューラ」という表現は注意が必要です。 cron のように任意の処理を実行する汎用スケジューラではなく、ダッシュボードの定期レポートや Grafana の通知・運用機能として理解します。

  • Grafana アラート: Zabbix のトリガーや Sentinel の分析ルールに近い。
  • スケジュールレポート: 毎週月曜に監視状況 PDF をメールで送る定期報告に近い。
  • Managed Identity: Azure VM から Key Vault へシークレットなしでアクセスする考え方と近い。
  • サービスプリンシパル: Terraform や GitHub Actions が Azure へログインするためのアプリ ID に近い。
  • Private Link: Storage Account を Private Endpoint 経由だけにする構成に近い。

KQLはSQLと同じなのか

KQL は SQL と似た目的で使うことはありますが、同じものではありません。 自分の場合は、SQL との比較よりも Linux コマンドのパイプとして理解した方が分かりやすかったです。

KQL は、ログを上から順番に絞って、整形して、集計する言語として理解します。 処理はパイプ | でつなぎます。

Linuxコマンドの例:

cat log.txt | grep error | sort

KQLの例:

Heartbeat
| where TimeGenerated > ago(6h)
| where OSType == "Linux"
| summarize Count=count() by bin(TimeGenerated, 15m), Computer
| order by TimeGenerated asc

Linux コマンドのパイプと同じように、KQL も | で処理をつなぎます。 ただし、横に長く書くのではなく、読みやすいように改行しています。 そのため、「パイプの形のまま改行したもの」と理解すると分かりやすいです。

KQLの基本操作

KQL の基本イメージは、次の流れです。

テーブルを読む
  ↓
条件で絞る
  ↓
必要な列だけ見る
  ↓
集計する
  ↓
並び替える

まず覚える操作は、次のあたりです。

  • where: grep に近い。条件で絞る。
  • project: cut に近い。表示する列を選ぶ。
  • order by: sort に近い。並び替える。
  • take: head に近い。件数を制限する。
  • summarize count(): uniq -c や awk に近い。件数を集計する。
  • extend: awk に近い。計算列を追加する。
  • contains: grep "文字列" に近い。文字列を含む行を探す。
  • bin(TimeGenerated, 15m): 時刻を15分単位に丸める。Grafana の時系列グラフで重要。

今回使ったKQLの分解

前回の記事で使った Windows 向けの Heartbeat クエリを分解します。

Heartbeat
| where TimeGenerated > ago(6h)
| where OSType == "Windows"
| summarize Count=count() by bin(TimeGenerated, 15m), Computer
| order by TimeGenerated asc
  • Heartbeat テーブルを読む。
  • 直近6時間に絞る。
  • Windows だけに絞る。
  • 15分ごと、Computer ごとに件数を数える。
  • 時刻の古い順に並べる。

Linux パイプ風に言えば、次のような感覚です。

cat Heartbeat
  | grep "直近6時間"
  | grep "Windows"
  | 15分単位で集計
  | sort

実際の Linux コマンドとしてこのまま動くわけではありませんが、頭の中のモデルとしてはかなり近いと感じました。

今回の結論

  • Azure Monitor dashboards with Grafana は、Azure Portal 内で使える SaaS 的な Grafana ビュー。
  • Azure Managed Grafana は、Azure 上に作る PaaS 的なマネージド Grafana。
  • 素の Grafana は、VM やコンテナに自分で構築する IaaS / 自前運用のイメージ。
  • Azure Monitor dashboards with Grafana は、Azure Monitor / Log Analytics など Azure 側の監視データを見る用途に向いている。
  • 外部データソース、Grafana アラート、スケジュールレポート、Private Link、Managed Identity などが必要になったら Azure Managed Grafana を検討する。
  • 今回のような Sentinel / Log Analytics の Heartbeat 確認なら、Azure Monitor dashboards with Grafana で十分。
  • KQL は SQL よりも、Linux コマンドのパイプ処理に近いと考えると理解しやすい。
  • KQL は、テーブルから始めて、where / project / summarize / order by をパイプでつないでいく。
