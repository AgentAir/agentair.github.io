Azure リソースは全部サブスクリプション移動できるわけではない：移動できるもの・作り直すものの整理

2026-05-18 / Azure / Resource Manager / Subscription / Migration

Home Articles About

はじめに

Azure で検証環境や本番環境を整理していると、 「このリソースグループを別サブスクリプションへ移動したい」 「検証用サブスクリプションから本番用サブスクリプションへ移したい」 という場面があります。

最初は、Azure Portal 上でリソースを選択して「移動」できるのであれば、 多くのリソースはそのまま別サブスクリプションへ移せるのではないかと思っていました。 しかし調べてみると、リソース種別ごとに移動可否が細かく分かれており、 単純な一括移動では済まないことが分かりました。

たとえば、VM は移動候補になりますが、DCR や Sentinel の設定は移動ではなく再作成前提で考える必要があります。 VNet は移動できますが、Peering や Private Endpoint が絡むと事前確認が必要です。 この記事では、Azure リソース移動の基本前提、移動できる代表例、移動できない代表例、実務上の考え方を整理します。

この記事で整理する「移動」の意味

ここで扱うのは、「サブスクリプションそのものを別のサブスクリプションへ移す」という話ではありません。 Azure リソースを、別リソースグループまたは別サブスクリプションへ移動する話です。

Azure Portal のリソースグループ画面などから実行できる「移動」は便利ですが、 すべてのリソースに対して同じように使えるわけではありません。 特に、監視、セキュリティ、ネットワーク境界に近いリソースは、 「移動」よりも「移動先で作り直す」「関連付けを再設定する」と考えたほうが安全なことが多いです。

Azure リソース移動の基本前提

まず、リソース種別ごとの可否を見る前に、Azure リソース移動の前提を押さえておきます。

  • 移動元・移動先サブスクリプションは、同じ Microsoft Entra テナント内である必要がある。
  • リソース移動はリージョン移動ではない。移動しても物理的なリージョンは変わらない。
  • 移動するとリソース ID が変わる。スクリプト、ダッシュボード、監視設定、外部管理台帳でリソース ID を参照している場合は更新が必要。
  • 移動中は移動元・移動先リソースグループにロックがかかる。移動中の作成、削除、更新はできない。
  • VM のように依存関係があるリソースは、VM / NIC / Disk / Public IP / VNet / NSG などをまとめて確認する必要がある。
  • 子リソースは単独で移動できないことがある。親リソースと一緒に動くのか、親が移動不可なのかを確認する。

実務では、Azure Portal の移動前検証だけに頼らず、対象リソースの種類、依存関係、公式ドキュメントの移動可否を先に確認しておくのが安全です。

移動できる代表例

まずは、移動候補として考えやすい代表例です。 ただし「可」と書いていても、依存リソースや構成によって追加作業が必要になることがあります。

種別 移動可否 注意点
Virtual Machine 可 NIC、Disk、VNet、Public IP、NSG、Backup 構成など依存リソースも確認する。
Managed Disk 可 Disk Access / Disk Encryption Set は不可。暗号化構成に注意する。
Snapshot 条件付き可 Full snapshot は移動候補。Incremental snapshot は制限がある。
NIC 可 VM とセットで確認する。関連する NSG や Public IP も見る。
VNet 可 Peering がある場合は事前に解除し、移動後に再設定する。
NSG 可 比較的移動しやすいが、サブネットや NIC への関連付けを確認する。
Route Table 可 関連サブネットとの依存関係を確認する。
Storage Account 可 リージョン移動ではない点に注意する。Private Endpoint や診断設定がある場合は別途確認する。
Key Vault 可 Disk Encryption 用途やアクセスポリシー、RBAC、Private Endpoint 連携に注意する。
Log Analytics Workspace 可 Automation Account 連携、Sentinel、有効化済みソリューション、診断設定の再確認が必要。
Recovery Services Vault 条件付き可 Backup / ASR / CMK 構成に注意。保護状態によって事前作業が必要になる。
Azure Arc machine 可 Microsoft.HybridCompute/machines は移動候補。拡張機能や DCR 関連付けは別途確認する。
App Service / Function App 条件付き可 Private Endpoint、VNet 統合、App Service Plan、証明書などの依存関係に注意する。
Azure SQL Server / Database 可 Server と Database の配置関係、Private Endpoint、診断設定、ファイアウォール設定を確認する。

移動できない／作り直し前提に近い代表例

次は、移動ではなく再作成・再設定を前提にしたほうがよい代表例です。 監視ルール、関連付け、セキュリティ境界、ネットワーク境界に近いものが多い印象です。

種別 移動可否
Managed Identity不可
Data Collection Rule / DCR不可
Diagnostic Settings不可
Metric Alert不可
Activity Log Alert不可
Microsoft Sentinel の Analytics Rule / Data Connector / Incident など不可
Application Gateway不可
Azure Firewall不可
Firewall Policy不可
NAT Gateway不可
VPN Gateway / Virtual Network Gateway不可
ExpressRoute Circuit / Gateway不可
Standard SKU Load Balancerサブスクリプション間移動不可
Azure Bastionサブスクリプション間移動は不可と考える
AKS managed cluster不可
Azure Container Instance不可
Databricks Workspace不可
Disk Access不可
Disk Encryption Set不可
Compute Gallery / Shared Image Gallery不可
SQL IaaS Agent extension resource不可
App Service Environment不可
Front Door Classic不可
Log Analytics Cluster不可

ここで大事なのは、「移動できない = 使えない」ではなく、 「移動先に同じ役割のリソースや設定を作り直す」と考えることです。 IaC、エクスポート、手順書、Portal の設定値メモを使って再現できる状態にしておくと、移動作業の見通しがよくなります。

特に注意するリソース

VM

VM は移動できますが、VM 単体だけを見て判断すると危険です。 実際には NIC、Disk、Public IP、NSG、VNet、Backup 構成などが絡みます。

  • Marketplace Plan 付き VM
  • Azure Disk Encryption 使用 VM
  • Azure Backup 保護中 VM
  • Availability Set 内 VM
  • Standard SKU Public IP が関連する VM

VM を移動候補にする場合は、Azure Portal の検証だけでなく、 対象 VM のネットワーク、ディスク、バックアップ、監視、拡張機能まで一覧化してから判断するのがよいです。

Private Endpoint

Private Endpoint はかなり注意が必要です。 Private Endpoint 自体が常に安全に移動できるわけではなく、 接続先の Private Link resource によって可否が変わります。

Microsoft Learn のネットワーク移動制限では、移動をサポートする private-link resource が列挙されています。 それ以外の Private Link resource は移動をサポートしないため、 実務上は Private Endpoint を「移動」ではなく「移動先で再作成」前提で考えるほうが安全です。

Sentinel / Monitor 系

Microsoft Sentinel、DCR、Diagnostic Settings、Alert などは、 リソースそのものというより「設定・関連付け・ルール」の要素が強いです。 そのため、次のように分けて考えます。

  • Log Analytics Workspace は移動候補。
  • Sentinel のルールやコネクタは再作成・再設定前提。
  • DCR は再作成前提。
  • Diagnostic Settings は再作成前提。
  • Alert Rule は再作成前提。

Sentinel を使っている環境では、Workspace だけを動かせば終わりではありません。 コネクタ、Analytics Rule、Automation Rule、Workbook、Alert、DCR、対象リソース側の診断設定まで含めて棚卸しする必要があります。

自分の検証環境に当てはめた整理

自分の検証環境に当てはめると、移動対象と作り直し対象は次のように分けて考えるのがよさそうです。

リソース 方針
VM移動候補。ただし NIC / Disk / VNet / NSG とセット確認。
VNet / Subnet / NSG移動候補。
Storage Account移動候補。
Bastion作り直し前提。
Private Endpoint基本は作り直し前提。
Private DNS Zone移動可だがリンク再確認。
Log Analytics Workspace移動候補。
Sentinel 設定作り直し前提。
DCR作り直し前提。
Azure Arc Machine移動候補。
AMA 拡張親リソース・DCR との依存確認。
Defender / Sentinel / Policy 系作り直し・再設定前提。

この整理にしておくと、「全部まとめて移動する」ではなく、 「移動する基盤リソース」と「移動先で再作成する運用設定」に分けて作業計画を立てられます。

実務上の結論

最終的には、次の3分類で考えるのが分かりやすいです。

移動してよいもの

  • VM
  • Disk
  • NIC
  • VNet
  • NSG
  • Storage Account
  • Key Vault
  • Log Analytics Workspace

ただし、依存関係は必ず確認します。 特に VM、VNet、Workspace は周辺リソースや関連付けが多いため、単体で判断しないほうがよいです。

移動しないほうがよいもの

  • Private Endpoint
  • Bastion
  • DCR
  • Diagnostic Settings
  • Sentinel 設定
  • Alert
  • Managed Identity
  • Firewall
  • Gateway 系

これらは移動可否が厳しかったり、移動できたとしても関連付けや状態の再確認が重くなりやすかったりします。 実務では、作り直しや再設定を前提にしたほうが作業計画を立てやすいです。

作り直し前提にするもの

  • 監視設定
  • セキュリティ設定
  • DCR
  • Sentinel
  • Private Endpoint
  • Bastion
  • Firewall
  • Gateway
  • AKS

監視・セキュリティ・境界系は、移動よりも再構築手順を整えるほうが安全です。 IaC で管理していない場合でも、設定値のスクリーンショットやエクスポート、Azure CLI の取得結果を残しておくと復元しやすくなります。

まとめ

Azure リソースは、Portal で「移動」メニューが見えるからといって、すべてをそのまま別サブスクリプションへ動かせるわけではありません。 VM、Disk、NIC、VNet、NSG、Storage Account、Key Vault、Log Analytics Workspace などは移動候補になりますが、 監視設定、DCR、Sentinel 設定、Private Endpoint、Bastion、Firewall、Gateway、AKS などは作り直し・再設定前提で考えたほうが安全です。

今回の学びは、「移動できるか」だけでなく、「移動後に同じ状態で運用できるか」まで見る必要があるという点でした。 本番環境や長く使う検証環境を整理するときは、移動対象、再作成対象、再設定対象を先に分けてから作業するのがよさそうです。

Azure の移動サポート状況は更新される可能性があります。最終判断は、必ず最新の Microsoft Learn、Azure Portal の移動前検証、実際の検証結果で確認してください。
