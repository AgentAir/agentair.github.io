SentinelのログをAzure Monitor dashboards with Grafanaで可視化してみた

2026-05-19 / Azure / Sentinel / Grafana / Log Analytics / Monitoring

Home Articles About

はじめに

Microsoft Sentinel に取り込んだログは、Sentinel の画面や Log Analytics Workspace から確認できます。 今回はそれに加えて、Grafana 形式のダッシュボードでも見やすく表示できるのかを検証しました。

ただし今回は、Azure Managed Grafana は作成していません。 料金をかけない検証にしたかったため、Azure Portal 内から利用できる Azure Monitor dashboards with Grafana を使いました。

この記事は、完成手順だけではなく、最初に迷った点や、実際に試して分かったことを整理した検証メモです。

最初の理解

最初に整理したのは、Grafana は Sentinel の代替ではない、という点です。 Sentinel はログを使って分析、検知、インシデント管理などを行う SIEM/SOAR 側のサービスです。 一方で Grafana は、データを見やすく並べるためのダッシュボードとして考えると理解しやすいと感じました。

Sentinel に入ったログの実体は、Log Analytics Workspace 上のログです。 そのため Grafana は Sentinel に直接つなぐというより、Sentinel が使っている Log Analytics Workspace に対して KQL を実行し、その結果をパネルとして表示するものだと理解しました。

つまり今回の位置づけは、「Sentinel でも確認できるデータを、別のダッシュボード画面で見やすく確認する」 というものです。

今回使った方式

今回は Azure Managed Grafana のリソースは作成せず、Azure Monitor dashboards with Grafana を使いました。 Azure Portal 内から Grafana 形式のダッシュボードを作成できる機能で、追加コストを避けたい検証には向いていると感じました。

ただし、Grafana 側を無料で使えたとしても、Log Analytics Workspace や Sentinel 側のログ取り込み費用は別問題です。 特に Syslog や SecurityEvent などを大量に取り込む場合は、可視化の前にログ取り込み量を意識しておく必要があります。

構成イメージ

Windows / Ubuntu / Azure Arc / AMA
    ↓
Log Analytics Workspace
    ↓
Microsoft Sentinel
    ↓
Azure Monitor dashboards with Grafana
    ↓
Grafana形式のダッシュボード表示

今回の検証では、Grafana の入口から Sentinel を選ぶというより、 Sentinel で使用している Log Analytics Workspace を選び、そこに入っているログを KQL で表示する流れになりました。

最初に誤解しやすい点

Grafana の画面を開いたとき、最初は Metrics を選ぶ画面に見えました。 しかし Sentinel のログを見たい場合は、Metrics ではなく Logs、Azure Monitor Logs、Log Analytics のようなログ参照側を選びます。

また、リソースとして仮想マシンを選ぶのではなく、Sentinel で使用している Log Analytics Workspace を選ぶ必要がありました。 VM ごとの絞り込みは Grafana のリソース選択で行うのではなく、KQL 側で Computer などを条件にして行うのが自然でした。

Heartbeatで接続を確認する

最初は Heartbeat を使って、Grafana から Log Analytics Workspace を参照できることを確認しました。 Heartbeat はエージェントの疎通確認に使いやすく、Windows と Linux の両方が見えるかを確認する入口として扱いやすかったです。 

Heartbeat
| where TimeGenerated > ago(6h)
| summarize Count=count() by bin(TimeGenerated, 15m), OSType
| order by TimeGenerated asc

この結果、OSType は Linux と Windows の両方で入っていることを確認できました。

ただし、最初は Grafana 上で線が 1 本だけに見えました。 テーブル表示で確認すると、Linux と Windows がそれぞれ同じ Count で出ていました。 つまり異常ではなく、Windows と Linux の Heartbeat 件数が同じだったため、2 本の線が完全に重なって 1 本に見えていただけでした。

WindowsとLinuxでパネルを分ける

重なって見える状態を避けるため、Windows と Linux でパネルを分けました。 パネル名は文字数制限があったため、短めにしています。

Windows パネル名: health-Win

Heartbeat
| where TimeGenerated > ago(6h)
| where OSType == "Windows"
| summarize Count=count() by bin(TimeGenerated, 15m), Computer
| order by TimeGenerated asc

Linux パネル名: health-linux

Heartbeat
| where TimeGenerated > ago(6h)
| where OSType == "Linux"
| summarize Count=count() by bin(TimeGenerated, 15m), Computer
| order by TimeGenerated asc

これで Windows と Linux を別々のパネルとして確認できるようになりました。 まずログが見えていることを確認し、その後で見た目を整える、という順番が良いと感じました。

Grafanaパネル設定の考え方

今回は最初から見た目を詰めすぎず、まず Log Analytics Workspace のログが正しく見えることを優先しました。 Heartbeat は Time series で表示し、Unit は none、Decimals は 0、Min は 0 にすると見やすかったです。

凡例は ON にして、どの Computer や OSType の線なのかを確認できるようにしました。 Auto refresh は短くしすぎず、検証では Off または 15m 程度で十分でした。 Heartbeat の件数グラフでは、最初からしきい値を無理に設定しない方が、データの見え方を確認しやすいと感じました。

Syslog検証で分かったこと

次に Ubuntu の Syslog も Grafana で表示しようとしました。 syslog-sev パネルを作りましたが、線は 1 本だけでした。 そこで Ubuntu 側で logger コマンドを使い、テストログを出しました。 

logger -p user.info "grafana-test-info"
logger -p user.warning "grafana-test-warning"
logger -p user.err "grafana-test-error"

しかし Grafana 側で次の KQL を実行しても、grafana-test のログは表示されませんでした。 

Syslog
| where TimeGenerated > ago(30m)
| where SyslogMessage contains "grafana-test"
| project TimeGenerated, Computer, Facility, SeverityLevel, ProcessName, SyslogMessage
| order by TimeGenerated desc

理由としては、以前 Sentinel を最小構成で作ったため、DCR に Syslog 収集設定が入っていない可能性が高いと判断しました。 今回の目的は Grafana から Sentinel / Log Analytics のログを見られることの理解だったため、Syslog 用の DCR 追加は今回は実施しませんでした。

今回の到達点

  • Azure Monitor dashboards with Grafana を開けた
  • Sentinel のログを見るには Log Analytics Workspace を選ぶべきだと理解した
  • Heartbeat を Grafana で表示できた
  • Windows / Linux を別パネルで表示できた
  • Syslog は DCR 側の収集設定が必要そうだと分かったため、今回は対象外にした

今回の学び

Grafana は Sentinel の代替ではなく、可視化用のダッシュボードとして使うものだと整理できました。 Sentinel で使うログは Log Analytics Workspace に保存されており、Grafana ではその Workspace を選んで KQL で表示します。

最初の確認には Heartbeat が向いていました。 一方で Syslog や SecurityEvent などは、そもそも DCR で収集されていなければ Grafana には表示されません。 可視化を考える前に、「Log Analytics にデータが入っているか」を確認することが重要だと分かりました。

今回は料金を避けるため Azure Managed Grafana は作らず、Azure Monitor dashboards with Grafana だけで検証しました。 小さく試して理解するには、この進め方がちょうど良かったです。
